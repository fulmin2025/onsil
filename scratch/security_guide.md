# Supabase 데이터 보안 및 해킹 방지 가이드

소중한 고객 정보와 장례식장 데이터를 해킹으로부터 안전하게 보호하기 위한 핵심 보안 전략을 정리해 드립니다.

## 1. 행 레벨 보안 (RLS) 적용 (가장 중요!)
이미 구현해 드린 **RLS(Row Level Security)**가 보안의 가장 기본이자 강력한 방어선입니다.
- **원리**: API 키(anon key)가 유출되더라도, 데이터베이스 자체에서 "본인 데이터가 아니면 접근 불가"라는 규칙을 강제합니다.
- **조치**: 새로 생성하는 모든 테이블에 RLS를 활성화하고, `SELECT true`와 같은 허술한 정책이 없는지 주기적으로 점검해야 합니다.

## 2. API 키 관리 및 노출 방지
- **Anon Key**: 프론트엔드(웹사이트)에서 사용하는 키입니다. RLS가 설정되어 있다면 노출되어도 안전하게 설계되어 있습니다.
- **Service Role Key**: 모든 보안을 우회하는 **'마스터 키'**입니다. 이 키는 절대로 자바스크립트 코드나 프론트엔드에 노출되어서는 안 됩니다. (오직 백엔드 서버나 Edge Functions에서만 사용해야 합니다.)

## 3. 강력한 인증 시스템 구축
- **다요소 인증 (MFA)**: 관리자 계정에는 반드시 MFA(OTP 등)를 설정하여 이메일/비밀번호가 유출되어도 로그인을 막아야 합니다.
- **이메일 확인**: 회원가입 시 실제 소유한 이메일인지 확인하는 절차를 활성화하세요.
- **비밀번호 정책**: 최소 8자 이상, 특수문자 포함 등 강력한 비밀번호 사용을 강제해야 합니다.

## 4. 민감 정보 암호화 (Encryption)
- Supabase는 저장 시(at rest) 기본적으로 암호화되지만, 더 높은 보안을 위해 주민등록번호나 계좌번호 같은 **극도로 민감한 정보**는 별도의 암호화 확장을 사용하거나 애플리케이션 레벨에서 암호화하여 저장할 수 있습니다.

## 5. 보안 감사 및 로그 (Audit Logs)
- Supabase의 **'Logs'** 메뉴를 통해 누가 언제 어떤 데이터에 접근했는지 모니터링하세요.
- 비정상적으로 많은 데이터를 한 번에 조회하는 시도가 있다면 즉시 차단해야 합니다.

## 6. 속도 제한 및 WAF 설정
- 단시간에 수만 번의 요청을 보내는 **브루트 포스(Brute-force)** 공격을 막기 위해 API 요청 속도 제한(Rate Limiting)을 설정해야 합니다. (Supabase 기본 제공 기능 활용)

## 7. 정기적인 백업
- 데이터가 손상되거나 랜섬웨어 공격을 받았을 때를 대비해 Supabase의 자동 백업 기능을 확인하고, 필요한 경우 주기적으로 데이터를 내보내어 별도의 안전한 장소에 보관하세요.

---
> [!TIP]
> **현재 적용된 상태**: RLS 정책을 통해 본인 데이터 외에는 조회가 불가능하도록 기초 공사가 완료되었습니다.
> **다음 단계**: 관리자 계정(`fulmin@nate.com` 등)의 비밀번호를 매우 강력하게 설정하시고, 가능하다면 Supabase 대시보드에서 MFA를 활성화하시는 것을 권장합니다.
